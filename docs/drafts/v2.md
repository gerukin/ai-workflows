# Workflow Specification

A workflow is a definition of steps to be executed by an AI agent in response to a trigger.

## Triggers

There is no definition for a trigger as it is the responsibility of the application to manage triggers. This includes deciding the kinds of triggers, when they are fired, and how they are configured. A simple trigger could be a user action, a scheduled event, or a system notification. At runtime, a trigger provides the following input to the workflow when starting it...

- Context: Triggers can provide initial context data to the workflow when it is started.
- How much: Triggers can specify constraints on time and cost budgets for the workflow execution.
- Who: Triggers can specify which agent the workflow is executed as.
- What bounds: Triggers can specify constraints on the tool parameters used in the workflow.

### Context

`trigger.context`: The context data provided by the trigger which started the workflow. This can include any relevant information about the event that caused the workflow to start. This is generally available to all steps in the workflow.

Context can be specified as markdown or YAML/JSON data via the `trigger.contextType` field.

> [!NOTE]
> JSON is always first automatically converted to YAML for easier reading by the LLM (with a comment indicating the conversion).
>
> While markdown (including markdown with FrontMatter) is supported, it is recommended to pass YAML/JSON for structured data as it may allow the LLM to later use JSONpath expressions to extract specific data more easily. Readability to an LLM should be similar for both formats.

#### Agent (who)

`trigger.agentContext`: Optional context describing the agent which is executing the workflow. This is provided similarly to normal trigger context, and can include information such as the agent's identity, permissions, and other relevant metadata. The agent can use this to understand its role and capabilities within the workflow.

> [!NOTE]
> Only system and built-in tools are executed by the framework directly. They do not depend on a specific agent persona.
>
> Application tools however are executed by the application. For tools which require authentication or specific permissions, the application must ensure that the agent has the necessary access rights, and uses the appropriate credentials when executing the tool on behalf of the agent.

> [!NOTE]
> The LLM to use for the workflow is passed by the application when starting the workflow execution and outside of the workflow definition itself. A future version may allow specifying preferred LLM types per step (ex: smart, fast, etc) via a map managed by the application.

### Constraints

`trigger.constraints.maxTotalTime`: The maximum time budget for the workflow execution, in seconds.

`trigger.constraints.maxExecutionTime`: The maximum time budget for the workflow execution, in seconds. This does not include time spent waiting for external events such as approvals or human input, but does include network and tool call time.

`trigger.constraints.maxCost`: The maximum cost budget for the workflow execution, in the base unit used by the app (for example money in USD cents or abstract compute units). The cost for each action is calculated by the app and provided to the framework at runtime.

`trigger.constraints.toolParameterBounds`: Optional constraints on tool parameters which can be used by the workflow. This is a map of tool IDs to overriden parameter constraints (as Zod schema).

> [!IMPORTANT]
> At this time constraints are hard constraints which terminate the workflow if exceeded (this is evaluated after each LLM or tool invocation). Future versions will try to introduce budget management skills and report budget usage to the agent.
>
> We will also consider providing ways for an app to expose remaining time and cost projections in the future (the app may for example calculate this by keeping track of averages or estimating fixed amounts), shared with the agent and understood via system skills.

### Trigger cascade

> [!WARNING]
> Workflows can take actions which may eventually result in other triggers firing. Some of these triggers may start the same or related workflows again, resulting in potentially infinite loops, or a cascade of workflow executions, which may or may not stop within a reasonable overall time or cost budget.
>
> Trying to control this problem is the responsibility of the application managing the triggers. Some possible strategies include limiting:
>
> - Concurrent workflow executions overall or per trigger.
> - Total workflow executions started by a given trigger within a time period.
> - Total time or cost budgets for agents or other entities (users, organizations, etc) per period.
>
> Any of these blunt strategies could however also limit or even severely hamper legitimate use cases. It is recommended to always allow LLMs to exit workflows early when they determine that no further actions are needed, and trigger workflows only when really needed.

## Steps

A step has instructions defined at workflow definition time, and may optionally specify skills to load for that step. Some steps can have branches or substeps (see [control flow](#control-flow) below).

LLMs must call tools to perform actions (including outputing results to humans). Text output from the LLM is only used inside the framework for context management and artifact generation (which are then possible to include in tool calls, or as context).

The output of a step is always a short summary of what was done in that step, which is used as context for downstream steps. The last summary of the last step is a very short summary of the entire workflow execution.

### Context

Context is managed conservatively to avoid overwhelming the LLM with too much data. Each step has access to:

- `trigger.context`: The context provided by the trigger which started the workflow.
- Summaries of previous steps: Each step has access to the summaries produced by all previous steps in the workflow. This allows the LLM to understand what has been done so far and build upon that knowledge.
- Artifacts produced by previous steps: If a previous step produced artifacts, those artifacts are available as references which can either be used in tool calls, or their content can be optionally loaded into context by the LLM as needed.
- Skills and tools (see [skills](#skills) below) for the current step.

Tool calls and results are only available inside the step which made them. They are not available to downstream steps directly, but their results can be summarized or included in artifacts which are then available to later steps.

> [!TIP]
> Generally steps should not be split into multiple small steps unless it really improves the LLM execution because the overall logic would otherwise be too complex or the context grow too large too quickly within a single step. Generally, simple actions on simple data should be done in a single step, leveraging the instructions and skills to let the LLM orchestrate itself and keep full context between small actions.
>
> Only break steps when running into complexity or context size issues.
>
> When split, costs can improve or worsen depending on the specific case.

### Step type

In `currentStep.type`, the following step types are supported:

- `normal`: A generic step with instructions and optional skills.
- `for-each-sequential`: A step that prompts the LLM to produce a list of items, and then executes substeps, for each item sequentially.
- `for-each-parallel`: A step that prompts the LLM to produce a list of items, and then executes substeps, for each item in parallel.
- `choose-branch`: A step that forces the LLM to choose one of several named branches to execute based on the context.
- `choose-branches-sequential`: A step that allows the LLM to choose multiple branches to execute based on the context, one after the other.
- `choose-branches-parallel`: A step that allows the LLM to choose multiple branches to execute based on the context, all in parallel.

> [!NOTE]
> Each step has its own summary but loops and branches also have overall summaries. This needs to be kept simple and efficient as loops can generate many many steps and summaries may become very large.

#### Branches

A branch is just a named substep inside a `choose-branch` or `choose-branches` step.

### Instructions

Each step has `currentStep.instructions` which guide the AI agent on what to do during that step. These instructions should be clear and concise, providing all necessary context and guidance for the agent to perform the required actions. They can include advice for calling tools or loading skills, the sort of output expected (considering other downstream steps), and any other relevant information. They could even include advice on how to exit early if needed (see [control flow](#control-flow) below).

> [!NOTE]
> Placeholders are not supported. Skills (which support some limited placeholders) and tools (which are open-ended) should be used to provide dynamic data to the agent instead. Referencing other steps must be done in natural language if needed.

### Control flow

LLMs can always choose to:

- End (or even skip) the current step and proceed to the next one.
- Loops:
  - End the current iteration of a for-each loop.
  - End the current for-each loop if more items are coming (if sequential).
- Branches:
  - End the current branch.
  - End the current branches flow if more branches are coming (if sequential).
- End the entire workflow early.

Instructions can be included in `instructions` to guide this behavior more explicitly.

> [!NOTE]
> An LLM is always able to end the current step by simply not performing any actions and outputting its summary as text. Other control flow actions are done via special tool calls (relevant skills are included automatically by the framework).
>
> It is not possible to disable control flow actions at this time. This could be considered in the future if needed.

## Skills

Skills are bundles of instructions and tools which can be made available to LLM agents as needed.

There are two kinds of skills:

- System skills: these are built-in skills which provide core functionality for the framework itself. They are prefixed with `_system/` in their IDs and are managed by the system.
- User skills: these are custom skills created by users or organizations to provide specific functionality.

Everything in the framework is a skill. Nothing is hardcoded except workflow execution itself. Even control flow is implemented via system skills and tools, loaded automatically as needed.

> [!NOTE]
> We will only focus on user skills in this documentation as system skills are mostly internal and not meant to be used or overridden by users at this time.
>
> The framework will also supply a handful of built-in skills for common tasks (ex: date and time handling, simply formatted math and conversions, planning, etc). They are functionally similar to user skills and can be included in workflows as needed (or ignored).

Each application provided user skill has:

- `id`: The unique identifier of the skill.
- `descriptionForAgent`: A brief description of the skill for the AI agent.
- `descriptionForHuman.{language code}`: An optional brief description of the skill for human understanding (not used by the LLM).
- `tools`: An optional array of tool IDs provided by the skill.
- `body`: Optional detailed description of the skill for the AI agent, which may include placeholders for dynamic data. When not provided, the tools are still made available (it is considered a tools only skill). No body and no tools is not allowed.

### Skill selection

Skills are specified at workflow definition time per step using `currentStep.skills` (array of skill objects). Each skill object includes:

- `id`: The unique identifier of the skill to load.
- `loaded`: Optional boolean indicating whether the skill should be preloaded before the step starts. Defaults to `false`.

Except for preloaded skills, which are available in full from the start of the step, other skills are loaded on demand by the LLM. The LLM only sees a list of currently available but unloaded skills, and can choose to load them as needed (using a special tool).

> [!NOTE]
> Agents decide to load unloaded skills based on their descriptions. Once loaded, all tools from that skill are available to the agent, and the body (if any) is included in context for the agent to understand how to use the skill.

> [!NOTE]
> System skills are always loaded automatically by the framework as needed and are used to provide all core functionality (ex: control flow, context management, etc).

### Tools

Tools are the primary way for AI agents to perform actions within a workflow. They are provided by skills and can be called by the LLM to execute specific tasks. Similar to skills, there are several kinds of tools:

- System tools: these are built-in tools that provide core functionality for the framework itself. They are prefixed with `_system_` in their IDs and are managed by the system skills, and run by the framework directly. Many things, like reading artifacts, managing control flow, etc, are done via system tools.
- Built-in tools: these are common tools provided by the framework for general-purpose tasks. They are executed by the framework directly but otherwise function similarly to application tools.
- Application tools: these are custom tools created by users or organizations to provide specific functionality. They are executed by the application managing the workflow.

> [!NOTE]
> For a discussion of permissions see [agent (who)](#agent-who) above.

LLMs can execute tools freely. Instructions can be included in `currentStep.instructions` to guide tool usage, but ultimately the LLM decides which tools to call, how often, and when. `workflow.constraints.toolParameterBounds` can be used to limit tool parameter values if needed (see [constraints](#constraints) above).

#### Tool execution

The application is responsible for executing application tools when called by the framework. Execution is timed and costed by the application, and the framework is informed of the results (see [constraints](#constraints) above for the metrics tools must return).

A tool result must also include markdown, text, or YAML/JSON data which is provided back to the LLM as context for further actions. If the result is structured data in JSON, the framework automatically converts it to YAML for easier reading by the LLM (with a comment indicating the conversion). This is then included in context for the LLM to use in the current step, and can be used in downstream steps via LLM generated summaries or artifacts.

#### Synchronous and asynchronous tools

Tools can return right away (near synchronous) or may take some time to complete (asynchronous). The framework handles both cases.

- Synchronous tools: These tools are awaited by the framework until they return a result.
- Asynchronous tools: These tools return a job ID immediately and the step execution stops temporarily. Step execution resumes later when the application informs the framework that the tool has completed (with the final result).
